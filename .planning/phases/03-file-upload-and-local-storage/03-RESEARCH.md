# Phase 3 Research: File Upload and Local Storage

## Scope

Phase 3 adds the local file intake layer for logistics documents. The backend should accept one or more uploaded files, validate technical constraints, store files locally under generated IDs, persist metadata in SQLite/SQLModel, and return file IDs that later phases can attach to Agent tasks.

## Existing Patterns

- `app/main.py` uses a FastAPI app factory and includes routers explicitly.
- `app/core/config.py` centralizes local settings through `pydantic-settings`.
- `app/db/session.py` provides a SQLModel engine and session dependency.
- `app/models/draft.py` shows the current SQLModel table style.
- `app/schemas/mock_kaihong.py` shows Pydantic response model style.
- `app/services/mock_kaihong.py` keeps business/storage helpers outside routers.
- `tests/test_mock_kaihong.py` uses `TestClient(app)` for end-to-end API coverage.

## Recommended Implementation Shape

| Concern | Recommendation |
|---------|----------------|
| Upload endpoint | One protected multipart endpoint for both single-file and multi-file uploads. |
| File types | Support JPEG, PNG, WEBP, and PDF by extension and content type. |
| Storage | Local project directory, generated IDs and stored filenames, no absolute paths in responses. |
| Metadata | SQLModel `UploadedFileRecord` table with file/user/linkage/status fields. |
| Hashing | Compute SHA-256 while saving or immediately after save for audit/debugging. |
| Linking | Store nullable `task_id` now; provide a small service helper and/or endpoint to associate a file with a future task ID without requiring a task table in this phase. |
| Auth | Reuse the existing mock bearer/current-user dependency so uploads are tied to `uploaded_by`. |

## Validation Boundary

Reject only technical problems:

- Unsupported type or extension.
- File larger than `20MB`.
- More than `10` files in a request.
- Total request payload larger than `100MB`.
- Empty or unsavable file.

Do not reject business-content problems in this phase:

- Wrong logistics document.
- Missing customer name.
- Unknown document type.
- Blurry or incomplete file.
- PDF that later extraction cannot parse.

Those are Phase 5/6 Agent and clarification concerns.

## Risks

- `UploadFile.content_type` can be client-provided, so planner should validate both MIME type and filename extension.
- Tests should avoid relying on real large files; use generated byte payloads.
- Local upload directories and uploaded files must be ignored by git.
- The existing app creates tables at import/startup, so new models must be imported before `SQLModel.metadata.create_all(engine)`.

## Research Complete

Phase 3 is straightforward and can be implemented as one focused execution plan.
