    - Single supported image upload returns one uploaded file metadata object.
    - Multi-file upload with image plus PDF returns two uploaded file metadata objects.
    - Unsupported file type is rejected.
    - Oversize file is rejected without saving metadata.
    - Too many files are rejected.
    - Response does not expose an absolute local filesystem path.

    Create `app/schemas/uploads.py` with Pydantic response models:
    - `UploadedFileResponse(file_id: str, original_filename: str, file_kind: str, content_type: str, size_bytes: int, status: str, customer_id: str | None = None, task_id: str | None = None, document_type: str | None = None, created_at: datetime)`
    - `UploadFilesResponse(files: list[UploadedFileResponse])`
    - `LinkFileTaskRequest(task_id: str)`

    Create `app/services/uploads.py` with helper functions and constants:
    - Allowed extensions/content types for JPEG, PNG, WEBP, and PDF.
    - `class UploadValidationError(ValueError)` carrying an HTTP-safe message.
    - `validate_upload_batch(files: list[UploadFile], settings: Settings) -> None`
    - `save_uploaded_files(session: Session, files: list[UploadFile], uploaded_by: str, customer_id: str | None = None) -> list[UploadedFileResponse]`
    - `get_uploaded_file(session: Session, file_id: str) -> UploadedFileResponse | None`
    - `link_file_to_task(session: Session, file_id: str, task_id: str) -> UploadedFileResponse | None`

    Service behavior:
    - Generate file IDs as `file_` plus a UUID hex prefix.
    - Generate stored filenames from file ID plus normalized extension.
    - Store files below `get_settings().upload_dir`.
    - Store only relative paths in `UploadedFileRecord.storage_path`.
    - Compute `sha256`.
    - Derive `file_kind` as `image` for image MIME types and `pdf` for PDFs.
    - Do not inspect or reject business document content.
    - If batch validation fails, do not create metadata records for any file in the batch.
  </action>
