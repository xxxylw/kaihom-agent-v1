# 凯鸿之翼手机物流订单录入 Agent 项目书

## 1. 项目背景

浙江凯鸿供应链的业务人员在处理物流订单时，常常需要根据托书、收据、提单、费用单、运输单证等材料手动录入订单信息。单证来源复杂，格式不统一，很多信息通过手机拍照、微信转发、客户邮件或纸质单据流转，人工录入容易出现效率低、字段遗漏、信息误填、重复沟通等问题。

本项目希望建设一个手机可访问的物流订单录入 Agent，让业务人员通过手机拍照上传单证，由 Agent 自动识别订单关键信息。如果字段缺失或识别置信度不足，Agent 主动追问业务人员补充。业务人员确认后，系统生成一份可录入凯鸿之翼的订单草稿，最终由业务系统保存或提交。

当前阶段尚未接触凯鸿之翼真实系统，也暂时拿不到真实接口、字段规则和鉴权方式。因此第一阶段不直接对接凯鸿之翼，而是先建设一套“模拟凯鸿之翼接口”的本地验证环境，把手机上传、单证识别、字段抽取、缺失追问、草稿生成、草稿保存的完整链路跑通。等后续拿到凯鸿之翼接口文档后，再把模拟接口替换为真实接口。

## 2. 项目目标

建设一个面向业务人员的移动端 AI 录单助手，核心目标如下：

1. 支持手机拍照上传托书、收据、提单、装箱单、费用单等物流单证。
2. 自动识别单证中的客户、地址、联系人、箱号、车牌、港口、仓库、费用、日期等关键信息。
3. 将识别结果转换成结构化订单草稿字段。
4. 对缺失字段、低置信度字段、冲突字段进行追问和确认。
5. 支持业务人员在手机端修改、补充、确认订单草稿。
6. 与凯鸿之翼系统对接，将确认后的草稿写入订单系统。
7. 保留上传文件、识别结果、追问过程、确认记录和接口调用日志，方便审计和追溯。

当前验证阶段的目标调整为：

1. 不依赖真实凯鸿之翼系统。
2. 用 Mock API 模拟凯鸿之翼的客户、订单、字典、草稿保存接口。
3. 用本地数据库保存模拟订单草稿。
4. 先验证 Agent 识别和追问流程是否可用。
5. 先产出一套未来可对接凯鸿之翼的接口契约。

## 3. 推荐技术路线

推荐采用“Java 主系统 + Python Agent 服务”的混合架构：

```text
手机端 / 凯鸿之翼前端
        ↓
凯鸿之翼 Java / Spring Boot 后端
        ↓ HTTP 内部调用
Python FastAPI Agent 服务
        ↓
OCR / 多模态模型 / RAG / 字段校验 / 追问逻辑
        ↓
返回结构化订单草稿
        ↓
Java 后端保存到凯鸿之翼
```

在尚未接触凯鸿之翼前，建议先采用“模拟凯鸿之翼适配层”：

```text
手机端测试页面
        ↓
模拟凯鸿之翼后端 Mock API
        ↓ HTTP
Python FastAPI Agent 服务
        ↓
OCR / 多模态模型 / 字段抽取 / 追问逻辑
        ↓
返回订单草稿 JSON
        ↓
Mock API 保存到本地数据库
```

这层 Mock API 的作用是提前定义未来和凯鸿之翼交互的接口形状。后续拿到真实凯鸿之翼接口后，优先替换 Mock API 内部实现，而不是重写 Agent。

### 为什么推荐 Python Agent 服务

该项目的核心难点不是普通 CRUD，而是单证理解、OCR、图片预处理、字段抽取、多轮追问和结构化生成。Python 在这些方向生态更成熟，适合快速搭建和迭代：

- OCR 和图片处理：OpenCV、Pillow、PaddleOCR、EasyOCR 等。
- 文档解析：PyMuPDF、pdfplumber、unstructured、docling 等。
- LLM 和 Agent：OpenAI SDK、LangChain、LlamaIndex、自研 Agent pipeline。
- 向量检索和 RAG：FAISS、Milvus、Qdrant、Chroma、Weaviate 等。
- 实验效率高，适合快速调整 prompt、字段 schema、追问规则和识别流程。

### Spring AI 的位置

如果凯鸿之翼本身是 Java / Spring Boot 系统，Spring AI 可以作为 Java 侧补充，但不建议第一版把复杂单证 Agent 全部放进 Spring AI。

Spring AI 更适合：

- 在 Java 业务系统内接入大模型。
- 调用已有 Java Service。
- 做简单问答、简单 RAG、业务助手。
- 把订单、客户、车辆、费用等 Java 服务暴露成 AI tool。

本项目推荐：

- Java / 凯鸿之翼：负责登录、权限、订单保存、业务数据、系统集成。
- Python / FastAPI：负责图片识别、字段抽取、Agent 追问、结构化草稿生成。

当前没有真实 Java 系统可接入时，可以先用 FastAPI 同时模拟两类服务：

- Agent 服务：负责 AI 识别和追问。
- Mock 凯鸿之翼服务：负责模拟用户、客户、订单、业务字典、草稿保存。

等真实系统可用时，再把 Mock 凯鸿之翼服务替换成 Java / Spring Boot 或真实凯鸿之翼接口。

## 4. 核心业务流程

### 4.1 业务人员上传单证

业务人员在手机端进入“AI 录单”页面，选择拍照或上传图片/PDF。

上传内容可能包括：

- 托书。
- 收据。
- 提单。
- 装箱单。
- 费用单。
- 运输单证。
- 微信截图或客户转发图片。

### 4.2 Java 后端接收并校验

凯鸿之翼后端负责：

- 校验用户是否登录。
- 校验用户是否有录单权限。
- 保存原始文件到私有对象存储。
- 生成文件 ID、任务 ID、会话 ID。
- 调用 Python Agent 服务发起识别任务。

### 4.3 Python Agent 识别和抽取

Python Agent 服务负责：

1. 图片预处理：旋转、裁剪、去噪、增强、压缩。
2. OCR 或多模态识别：识别文本、表格、印章、票据字段。
3. 字段抽取：从文本中抽取订单字段。
4. 字段归一化：统一日期、地址、港口、箱号、车牌、费用格式。
5. 字段校验：检查必填字段、格式错误、业务冲突。
6. 置信度评估：标记哪些字段可信、哪些字段需要确认。
7. 追问生成：根据缺失字段生成业务人员能看懂的问题。

### 4.4 Agent 追问补全

如果信息不完整，Agent 不直接生成最终草稿，而是追问：

- “请补充联系人电话。”
- “提货时间未识别到，请选择或输入预计提货时间。”
- “识别到两个目的地址，请确认使用哪一个。”
- “箱号识别置信度较低，请确认是否为 XXXX1234567。”

业务人员回答后，Agent 更新订单草稿。

### 4.5 生成订单草稿

当必要字段齐全后，Agent 返回结构化 JSON，例如：

```json
{
  "draft_id": "draft_20260507_001",
  "customer_name": "某某贸易有限公司",
  "contact_name": "张三",
  "contact_phone": "138****0000",
  "pickup_address": "浙江省嘉兴市...",
  "delivery_address": "宁波舟山港...",
  "container_no": "XXXX1234567",
  "vehicle_no": "浙A12345",
  "pickup_time": "2026-05-08 09:00:00",
  "cargo_name": "家具配件",
  "fees": [
    {
      "fee_type": "运输费",
      "amount": 1200,
      "currency": "CNY"
    }
  ],
  "missing_fields": [],
  "low_confidence_fields": [],
  "source_files": [
    {
      "file_id": "file_001",
      "file_name": "托书照片.jpg"
    }
  ]
}
```

### 4.6 人工确认和写入凯鸿之翼

业务人员在手机端确认草稿后，Java 后端调用凯鸿之翼订单保存接口：

- 保存为订单草稿。
- 或提交到正式订单流程。
- 或进入人工审核队列。

建议第一版只生成“草稿单”，不要直接生成正式订单。这样更安全，也便于业务验证。

- 车牌号。

- 是否必填。

## 7. 接口设计建议

当前阶段建议把接口分成两组：

- Agent API：真正由 Python Agent 实现，负责识别、追问和生成草稿。
- Mock Kaihong API：模拟凯鸿之翼，负责提供业务数据和保存草稿。

### 7.1 Java 调 Python

创建识别任务：

```http
POST /agent/order-draft/tasks
```

请求示例：

```json
{
  "task_id": "task_001",
  "user_id": "u_123",
  "tenant_id": "kaihong",
  "session_id": "s_001",
  "files": [
    {
      "file_id": "file_001",
      "file_url": "https://private-signed-url"
    }
  ],
  "business_context": {
    "customer_id": "c_001",
    "source": "mobile"
  }
}
```

查询任务状态：

```http
GET /agent/order-draft/tasks/{task_id}
```

提交追问回答：

```http
POST /agent/order-draft/tasks/{task_id}/answers
```

生成最终草稿：

```http
POST /agent/order-draft/tasks/{task_id}/finalize
```

### 7.2 Python 调 Java

建议提供内部 API：

```http
GET /internal/customers/{customer_id}
GET /internal/orders/recent?customer_id=xxx
GET /internal/dictionaries/ports
GET /internal/dictionaries/warehouses
POST /internal/order-drafts
```

这些接口必须做服务间认证和权限控制。

### 7.3 当前阶段 Mock API

在拿不到凯鸿之翼真实接口前，先实现以下模拟接口。

模拟登录：

```http
POST /mock/auth/login
```

获取当前用户：

```http
GET /mock/users/me
```

获取客户列表：

```http
GET /mock/customers
```

获取客户详情：

```http
GET /mock/customers/{customer_id}
```

获取最近历史订单：

```http
GET /mock/orders/recent?customer_id=xxx
```

获取业务字典：

```http
GET /mock/dictionaries/ports
GET /mock/dictionaries/warehouses
GET /mock/dictionaries/fee-types
```

保存订单草稿：

```http
POST /mock/order-drafts
```

查询订单草稿：

```http
GET /mock/order-drafts/{draft_id}
```

这些接口先使用本地 JSON 文件或 SQLite / PostgreSQL 模拟数据。未来对接真实凯鸿之翼时，保留接口契约，替换数据来源。

## 8. 安全设计

### 8.1 用户认证

用户登录仍然走凯鸿之翼原有认证系统。Python Agent 不直接面对未认证用户。

推荐流程：

```text
手机端登录凯鸿之翼
        ↓
Java 后端验证用户身份
        ↓
Java 带可信用户上下文调用 Python Agent
```

### 8.2 服务间认证

Java 和 Python 之间必须有认证机制：

- 内网部署。
- HTTPS。
- API Key 或 JWT。
- 请求签名。
- 成熟阶段可使用 mTLS。

### 8.3 数据权限

Python Agent 不应该绕过 Java 业务权限。

原则：

- Python 不直接连接凯鸿之翼生产数据库。
- Python 通过 Java 内部 API 获取被授权的数据。
- Java 根据用户、部门、角色、客户范围过滤数据。
- Agent 只能拿当前任务需要的数据。

### 8.4 文件安全

单证文件可能包含客户、地址、费用、合同、票据等敏感信息。

要求：

- 文件私有存储。
- 访问链接短期有效。
- 不在日志中打印完整文件内容。
- 文件传输使用 HTTPS。
- 文件删除和归档有明确策略。

### 8.5 日志和审计

需要记录：

- 谁上传了文件。
- 上传了哪些文件。
- Agent 识别出了哪些字段。
- 哪些字段被人工修改。
- Agent 调用了哪些内部接口。
- 最终生成了什么草稿。
- 谁确认并提交。

日志中应脱敏：

- 手机号。
- 身份证。
- 银行账号。
- 详细地址。
- 合同金额。
- 客户敏感备注。

## 9. MVP 分期计划

### 第零期：无凯鸿之翼接口的本地模拟版

目标：在没有真实凯鸿之翼系统的情况下，先把完整产品链路做出来。

范围：

- 本地手机端或网页端上传单证。
- Mock 用户登录。
- Mock 客户、订单、港口、仓库、费用字典。
- Python Agent 识别单证字段。
- Agent 判断缺失字段并追问。
- 生成订单草稿 JSON。
- 保存草稿到本地数据库。
- 提供 Swagger / OpenAPI 文档，明确未来真实接口契约。

暂不做：

- 对接真实凯鸿之翼。
- 写入正式订单。
- 真实企业权限体系。
- 复杂审批流程。
- 生产级高可用部署。

### 第一期：可用验证版

目标：在第零期基础上，用少量真实脱敏样本证明拍照上传到生成订单草稿的闭环可行。

范围：

- 手机端上传图片。
- Python OCR / 多模态识别。
- 抽取 10 到 15 个核心字段。
- 缺失字段追问。
- 生成订单草稿 JSON。
- 人工确认后保存草稿。

暂不做：

- 自动提交正式订单。
- 复杂费用规则。
- 大规模 RAG。
- 多角色复杂权限。
- 高级报表。

### 第二期：业务增强版

目标：提高准确率，接入更多业务规则。

范围：

- 接入客户档案、历史订单、港口仓库字典。
- 相似历史订单参考。
- 客户简称和地址纠错。
- 字段置信度展示。
- 草稿修改记录。
- 审核流程。

### 第三期：生产完善版

目标：稳定上线和规模化使用。

范围：

- 完整权限控制。
- 服务监控。
- 失败重试。
- 队列异步处理。
- 模型效果评估。
- 成本统计。
- 数据脱敏。
- 多租户或多部门隔离。

## 10. 技术选型建议

### 后端

- Java / Spring Boot：凯鸿之翼主业务系统。
- Python / FastAPI：Agent 服务。
- Uvicorn / Gunicorn：Python 服务部署。

### AI 能力

- OCR：PaddleOCR、云 OCR、或多模态大模型。
- LLM：根据企业合规要求选择私有化模型或云模型。
- Agent 编排：第一版建议自研轻量流程，不必过早引入复杂框架。
- RAG：第二期再加入向量库和历史订单检索。

### 存储

- PostgreSQL / MySQL：任务、草稿、日志、字段结果。
- Redis：任务状态、缓存、限流。
- OSS / S3 / MinIO：文件存储。
- Milvus / Qdrant / FAISS：向量检索，第二期使用。

### 部署

- Docker 部署 Java 和 Python 服务。
- 内网服务互通。
- Nginx / API Gateway 管理入口。
- 生产环境使用 HTTPS。

## 11. 关键风险

### 11.1 识别准确率风险

拍照质量、单证格式、手写字、图片反光都会影响识别效果。

应对：

- 上传时提示拍照规范。
- 做图片预处理。
- 引入置信度。
- 必填字段必须人工确认。

### 11.2 业务字段不统一

不同客户、不同业务线的字段名称和录入规则可能不同。

应对：

- 先聚焦一个业务线。
- 先定义最小字段集。
- 建立字段字典和客户规则。

### 11.3 系统集成风险

凯鸿之翼是否有可用接口、字段是否完整、权限如何传递，都需要确认。

应对：

- 当前阶段先使用 Mock API，不阻塞 Agent 原型开发。
- 提前定义接口契约，降低未来切换成本。
- 优先对接草稿接口。
- 不直接写正式订单。
- Java 侧保留人工确认和审核。

### 11.4 数据安全风险

单证、客户、费用、地址都属于敏感业务数据。

应对：

- 服务间认证。
- 文件私有存储。
- 日志脱敏。
- 数据最小化传递。
- 明确模型供应商和数据使用策略。

## 12. 验收标准

第零期本地模拟版可按以下标准验收：

1. 不依赖真实凯鸿之翼系统即可运行。
2. 手机端或网页端可以上传单证图片。
3. Mock API 可以返回客户、历史订单、业务字典等模拟数据。
4. Python Agent 可以输出结构化订单草稿。
5. Agent 能识别缺失字段并生成追问。
6. 用户补充信息后，草稿字段可以更新。
7. 草稿可以保存到本地模拟订单表。
8. OpenAPI 文档能展示所有 Agent API 和 Mock API。

第一期 MVP 可按以下标准验收：

1. 手机端可以上传至少 3 类常见单证。
2. 系统能生成结构化订单草稿。
3. 核心字段识别准确率达到业务可试用水平。
4. 必填字段缺失时，Agent 能主动追问。
5. 业务人员可以修改并确认字段。
6. 草稿可以保存到凯鸿之翼或测试订单表。
7. 每次识别都有任务记录、文件记录、字段结果和操作日志。
8. 未登录用户不能访问录单功能。
9. Python Agent 不能绕过 Java 权限直接访问生产数据。

## 13. 需要向凯鸿之翼确认的问题

落地前需要收集以下信息：

1. 凯鸿之翼是否已有订单草稿接口。
2. 订单正式字段清单和必填规则。
3. 客户、地址、港口、仓库、费用等基础数据接口。
4. 用户登录和权限体系。
5. 文件上传和附件管理方式。
6. 是否允许单证图片发送给外部大模型或 OCR 服务。
7. 数据保存周期和审计要求。
8. 是否需要支持多部门、多客户数据隔离。
9. 订单草稿到正式订单的审核流程。
10. 第一批试点业务线和样本单证类型。

在尚未接触凯鸿之翼前，可以先自行假设一版字段和接口，但所有假设都要在文档中标记为“待真实接口确认”。后续拿到真实接口文档后，需要做一次字段映射表：

```text
Agent 草稿字段 → 凯鸿之翼真实订单字段
```

例如：

```text
customer_name → customerName
pickup_address → pickupAddress
delivery_address → deliveryAddress
container_no → containerNo
```

## 14. 推荐第一步

建议先做一个小范围试点：

1. 先定义一版模拟订单字段 schema。
2. 准备 10 到 20 张公开样例或自制模拟单证图片。
3. 用 FastAPI 做 Agent API 和 Mock Kaihong API。
4. 用 SQLite 或本地 JSON 保存模拟客户、字典和订单草稿。
5. 做一个手机可访问的简单上传页面。
6. 完成“上传文件 → 识别字段 → 追问缺失信息 → 生成草稿 → 保存草稿”的闭环。
7. 输出 OpenAPI 文档，作为未来和凯鸿之翼沟通的接口草案。
8. 后续拿到凯鸿之翼资料后，再替换 Mock API 数据源和字段映射。

## 15. 总结

该项目建议采用 Python FastAPI 作为 Agent 服务，负责单证识别、字段抽取、多轮追问和订单草稿生成；凯鸿之翼 Java 后端负责用户认证、权限控制、业务数据、文件管理和订单落库。

这种架构能发挥 Python 在 AI、OCR、文档处理和 Agent 编排上的优势，同时保留凯鸿之翼现有 Java 系统在业务规则、权限、安全和数据一致性上的优势。第一阶段应以“生成可确认的订单草稿”为目标，而不是直接自动创建正式订单，这样更容易落地，也更符合企业业务安全要求。

在当前尚未接触凯鸿之翼的情况下，最合理的路线是先做 Mock 版。Mock 版不是临时玩具，而是未来正式对接前的接口契约验证、业务流程验证和 Agent 效果验证。只要一开始把 Mock 接口边界设计清楚，后续替换成真实凯鸿之翼接口时，整体架构不需要推倒重来。
