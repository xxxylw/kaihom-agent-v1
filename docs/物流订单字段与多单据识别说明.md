# 物流订单字段与多单据识别说明

**适用项目：** 凯鸿之翼手机物流订单录入 Agent
**阶段定位：** v1 Mock 系统的业务字段参考和后续识别/追问依据
**结论摘要：** 物流订单不是一张固定表单。实际 ToB 物流委托通常由多张单据共同描述一次或多次运输任务，系统需要支持“多单据输入 -> 多订单/多段运输草稿 -> 缺失字段追问”。

## 1. 为什么不能只按一张订单做

客户找凯鸿这类 ToB 供应链/物流服务商时，通常不是只提供一张完整订单，而是提供一组业务材料：

- 托书或委托单：说明客户想让物流公司做什么。
- 商业发票：说明货值、币种、交易双方、品名。
- 装箱单：说明件数、包装、毛重、净重、体积、箱规。
- 提单/海运订舱资料：说明 Shipper、Consignee、Notify、港口、船名航次、箱号封号。
- 空运单/AWB：说明机场、航空公司、AWB 号、件数、重量、计费重。
- 报关资料：说明 HS 编码、原产国、监管方式、运输方式、成交方式等。
- 回单/收据/POD：说明签收、费用、异常、回单要求。

所以我们的 Agent 不应该假设“上传一张图 = 一个订单”。更合理的是：

```text
一个 Agent 任务
  -> 多张单据图片
  -> 识别出多个文档类型
  -> 抽取多个字段片段
  -> 合并成一个或多个订单草稿
  -> 对缺失/冲突字段追问业务人员
```

## 2. 通用物流订单字段

这些字段基本适用于国内、国际、海运、空运、陆运：

| 模块 | 字段 |
|---|---|
| 客户与委托 | 委托客户、业务联系人、电话、客户订单号、合同/报价编号、紧急程度、结算主体、付款方 |
| 收发货方 | 发货人、收货人、通知方、实际提货联系人、实际送货联系人、报关委托方 |
| 货物信息 | 中文品名、英文品名、规格型号、件数、包装方式、毛重、净重、体积、尺寸、货值、币种、原产国、HS 编码 |
| 路线信息 | 起运地、目的地、提货地址、送货地址、起运港、目的港、始发机场、目的机场、仓库、堆场、口岸 |
| 时间要求 | 提货时间、送达时间、截关时间、截单时间、开船/航班时间、预计到达时间 |
| 贸易与费用 | 国内/国际、进口/出口、成交方式 Incoterms、运费预付/到付、保险、代垫费用、开票要求 |
| 操作要求 | 是否报关、是否仓储、是否派车、是否代办保险、是否需要回单、装卸要求、异常备注 |
| 单证材料 | 托书、发票、装箱单、合同、报关委托书、提单、运单、原产地证、许可证、危包/鉴定资料 |

## 3. 国内陆运/仓配字段

国内陆运和仓配更关注地址、车辆、时间窗、回单：

| 模块 | 字段 |
|---|---|
| 提送货 | 提货地址、送货地址、提货联系人、送货联系人、联系电话、装货时间窗、卸货时间窗 |
| 车辆 | 车型、车长、厢式/平板/冷藏、是否尾板、是否需要进厂预约、车牌号、司机电话 |
| 货物 | 件数、包装、重量、体积、是否超长、是否超重、是否易碎、是否温控 |
| 回单 | 签收单、POD、是否原件回单、回单照片、异常照片、代收货款 |
| 费用 | 干线费、提货费、送货费、装卸费、等待费、压车费、异常费用 |

典型追问：

- “送货地址缺少联系人电话，是否补充？”
- “货物体积缺失，是否按长宽高计算？”
- “客户是否要求原件回单？”

## 4. 国际海运字段

海运更关注订舱、箱、港口、提单：

| 模块 | 字段 |
|---|---|
| 基础角色 | Shipper、Consignee、Notify Party、订舱代理、船公司 |
| 船务 | Booking No.、B/L No.、船名航次、起运港 POL、卸货港 POD、目的地、截单时间、截关时间 |
| 箱务 | 整箱/拼箱、箱型箱量、箱号、封号、SOC/COC、提箱地、还箱地 |
| 货物 | 唛头、包装、件数、中文品名、英文品名、毛重、体积 CBM、危险品信息 |
| 提单 | 正本提单、电放、Sea Waybill、提单份数、运费预付/到付 |
| 报关 | HS 编码、报关品名、成交方式、监管方式、原产国、报关单证 |

典型追问：

- “目的港识别为 Los Angeles，但单据里又出现 Long Beach，是否确认？”
- “箱号有了，但封号缺失，是否后补？”
- “提单方式是正本、电放还是 Sea Waybill？”

## 5. 国际空运字段

空运更关注机场、航班、计费重、安检和时效：

| 模块 | 字段 |
|---|---|
| 空运单 | AWB 号、航空公司、始发机场、目的机场、代理信息 |
| 货物 | 件数、毛重、尺寸、体积重、计费重、货物描述、HS 编码、货值 |
| 时效 | 普货/快件/急件、预计航班、最晚交仓时间、预计到达时间 |
| 合规 | 是否带电、是否磁性、是否危险品、是否温控、是否木箱/托盘、是否需要鉴定 |
| 费用 | 空运费、燃油附加费、安检费、提货费、派送费、目的港费用 |

典型追问：

- “图片里有电池字样，是否属于带电货？”
- “尺寸已识别，但计费重需要系统计算，是否确认长宽高单位为 cm？”
- “目的机场代码不清楚，是 LAX 还是 SFO？”

## 6. 国际报关/贸易字段

国际业务通常需要报关或贸易单证字段。中国海关报关单填制规范中包含收发货人、口岸、运输方式、运输工具名称、航次号、提运单号、申报单位、监管方式等栏目。商业发票通常要支撑货物描述、货值、原产国、数量、编码等信息。

| 模块 | 字段 |
|---|---|
| 主体 | 境内收发货人、境外收发货人、消费使用单位/生产销售单位、申报单位 |
| 运输 | 进口/出口口岸、运输方式、运输工具名称、航次号、提运单号 |
| 贸易 | 监管方式、征免性质、成交方式、合同协议号、贸易国/地区 |
| 商品 | 商品编号 HS Code、中文品名、英文品名、规格型号、数量、单位、单价、总价、币制、原产国 |
| 单证 | 商业发票、装箱单、合同、报关委托书、原产地证、许可证、检验检疫材料 |

典型追问：

- “缺少 HS 编码，是否由业务员填写或从历史订单引用？”
- “发票币种是 USD，但托书未写成交方式，是否为 FOB/CIF/EXW？”
- “一票货物对应多个提单/运单时，是否需要拆成多票？”

## 7. 多单据类型识别清单

Agent 应该先判断每张图片属于哪类单据，再决定抽取哪些字段：

| 单据类型 | 识别线索 | 主要抽取字段 |
|---|---|---|
| 托书/委托单 | 委托、托运、订舱、客户指令 | 客户、路线、服务类型、提送货要求、特殊备注 |
| 商业发票 Commercial Invoice | Invoice、卖方、买方、金额、币种 | 发票号、交易双方、货值、币种、品名、数量、原产国 |
| 装箱单 Packing List | Packing、件数、毛重、净重、尺寸 | 件数、包装、重量、体积、箱数、唛头 |
| 海运提单/订舱单 B/L/Booking | Shipper、Consignee、POL、POD、Vessel | 船名航次、港口、箱号封号、提单方式、货描 |
| 空运单 AWB | AWB、Airport、Flight、Airline | AWB 号、机场、航空公司、件数、重量、计费重 |
| 报关资料 | 报关单、商品编号、监管方式、运输方式 | HS 编码、报关品名、监管方式、运输方式、提运单号 |
| 回单/收据/POD | 签收、收据、回单、费用 | 签收人、签收时间、费用、异常、回单要求 |

## 8. 多订单/多票拆分规则

一组图片里可能包含多个订单。建议后续 Agent 按这些信号拆分：

| 拆分信号 | 说明 |
|---|---|
| 不同客户订单号 | 不同客户订单号通常应拆为不同订单草稿。 |
| 不同收货人/目的地 | 同一客户但不同目的地，可能拆成多票运输。 |
| 不同提单/运单号 | 海运/空运/报关里常见，一票货物对应多个单号时要拆分或提示确认。 |
| 不同箱号/车次/航班 | 可能代表多票或多段运输。 |
| 不同运输方式 | 同一委托可能包含陆运+海运+派送，应作为多段运输记录。 |
| 金额/发票号不同 | 多张发票可能对应多票商品或多个订单。 |

建议草稿结构：

```json
{
  "task_id": "task_001",
  "customer_id": "cust_001",
  "source_documents": [
    {
      "document_id": "doc_001",
      "document_type": "commercial_invoice",
      "filename": "invoice.jpg",
      "confidence": 0.92
    }
  ],
  "order_items": [
    {
      "order_index": 1,
      "transport_mode": "ocean",
      "route_type": "international_export",
      "fields": {
        "shipper": "ABC Trading Co.",
        "consignee": "XYZ Imports LLC",
        "pol": "Ningbo",
        "pod": "Los Angeles"
      },
      "missing_fields": ["hs_code", "bl_release_type"]
    }
  ],
  "questions": [
    {
      "field": "hs_code",
      "question": "缺少 HS 编码，是否补充或从历史订单引用？"
    }
  ]
}
```

## 9. 对当前项目阶段的影响

### Phase 2 是否已经支持

原 Phase 2 计划支持：

- Mock 登录/当前用户。
- Mock 客户、字典、近期订单。
- 本地保存和读取草稿。
- 草稿 payload 是灵活 `dict`，理论上可以存多单据、多订单数据。

但原 Phase 2 没有明确支持：

- source documents 列表。
- order items 列表。
- document type 字典。
- transport mode / route type 字典。
- 多单据/多订单保存测试。

所以结论是：

```text
原 Phase 2 可以“装下”这些数据，但没有把它变成明确合同。
需要更新 Phase 2，让 Mock draft contract 明确支持多单据、多订单草稿。
真正的图片上传和自动识别仍然属于 Phase 3 和 Phase 5。
```

### 推荐阶段边界

| 阶段 | 应做什么 |
|---|---|
| Phase 2 | Mock Kaihong API 支持保存多单据、多订单草稿结构；字典包含单据类型、运输方式、路线类型。 |
| Phase 3 | 上传多张图片，给每张图片生成 file/document reference。 |
| Phase 4 | Agent task 维护 source_documents、order_items、questions、events。 |
| Phase 5 | Mock OCR/识别：判断单据类型，抽取字段，拆分多订单。 |
| Phase 6 | 缺失字段追问、用户回答、合并成最终草稿。 |
